# アンチパターン・バッドノウハウ集

プロジェクト憲章とプロジェクト計画書の運用において、実際の現場でよく見られる失敗パターンとその対策をまとめます。

<br>

## プロジェクト憲章のアンチパターン

### 1. 「とりあえず承認」症候群

**症状:**
- 目的や期待効果が曖昧なまま、「まずは予算だけ確保」という発想で憲章を作成
- スポンサーも内容を精査せず形式的に承認
- 「詳細は後で詰める」と先送りにする

**典型例:**
```
【悪い例】
目的: 「業務効率化のため」
期待効果: 「生産性向上」
スコープ: 「検討中」
```

**なぜダメか:**
- プロジェクト中盤で「そもそも何のためにやるのか」で揉める
- 成功判定ができず、プロジェクトが終わらない
- スコープクリープが止まらない

**対策:**
- 承認前に必ずSMART基準でチェック
- 「測定できない目標は承認しない」ルールの徹底
- スポンサーに「この目標で投資判断できますか?」と確認

---

### 2. コピペ憲章

**症状:**
- 過去の別プロジェクトの憲章をコピーして、プロジェクト名だけ変更
- 前提条件や制約条件が前プロジェクトのまま
- リスク一覧が使い回しで、今回のプロジェクト固有リスクが抜けている

**典型例:**
```
【悪い例】
前提条件: 「〇〇システムが2023年3月に稼働済みであること」
（実際は別システムで、該当しない）

主要リスク: 「海外拠点との時差によるコミュニケーション遅延」
（実際は国内のみのプロジェクト）
```

**なぜダメか:**
- 無関係な前提条件により、後で「聞いてない」が発生
- 本当に重要なリスクが見落とされる
- 関係者が「ちゃんと読んでいない」ことが露呈

**対策:**
- テンプレートは使ってもよいが、全項目を現プロジェクト用に書き直す
- ピアレビューで「これ前のプロジェクトの内容では?」をチェック
- 憲章レビュー会議で各項目を音読して確認

---

### 3. 全部入れ憲章

**症状:**
- 「念のため」とスコープを広げすぎる
- Out-of-Scopeを明記せず、「やるかも」をすべて含める
- 結果、10ページ以上の大作になる

**典型例:**
```
【悪い例】
スコープ:
- 販売管理システム刷新
- 在庫管理システム刷新
- 会計システム刷新
- 人事システム刷新（将来的に）
- データ分析基盤構築（可能であれば）
- AI活用検討（トレンドなので）
```

**なぜダメか:**
- 予算・期間が現実離れする
- 本当に必要なものが埋もれる
- 「将来的に」「可能であれば」が義務のように扱われる

**対策:**
- 憲章は1-2ページに収める鉄則を守る
- Out-of-Scopeを明記する勇気を持つ
- 「Phase 1で実施」と「Phase 2以降で検討」を明確に分ける

---

### 4. PMに丸投げ憲章

**症状:**
- スポンサーが憲章作成をPMに全部委任
- PM自身が目的や期待効果を「想像」で書く
- スポンサーは最後に署名だけする

**典型例:**
```
【悪い例】
作成プロセス:
1. スポンサー「とりあえずプロジェクトチャーター作っておいて」
2. PM「目的って何ですか?」
3. スポンサー「そっちで考えて。いい感じで」
4. PM、仕方なく自分で想像して記載
5. スポンサー、内容を読まずに承認
```

**なぜダメか:**
- スポンサーのコミットメントが得られない
- 後で「そんなつもりじゃなかった」と言われる
- PMが経営判断を勝手にすることになる

**対策:**
- 憲章はスポンサー起案が原則であることを明文化
- PMはドラフト作成を支援するが、内容はスポンサーが決定
- 承認会議でスポンサー自身に説明させる

---

### 5. 権限不明確問題

**症状:**
- PMの権限範囲が書かれていない
- 「調整して決めてください」だけで、決裁権限が不明
- 予算執行の上限が書かれていない

**典型例:**
```
【悪い例】
PM権限: 「プロジェクト全般の推進」
→ 具体的に何を決められるのか不明

予算: 「総額5000万円」
→ PMが単独で執行できる上限は?
→ 追加承認が必要な閾値は?
```

**なぜダメか:**
- 意思決定のたびにスポンサーへエスカレーション
- 判断が遅れてプロジェクトが停滞
- PM が責任を取れない構造になる

**対策:**
```
【良い例】
PM権限:
- 単独決裁: 50万円以下の追加費用、1週間以内のスケジュール調整
- スポンサー承認必要: 上記を超える変更、スコープ変更
- ステアリングコミッティ承認必要: 予算10%超過、3か月超の遅延
```

## プロジェクト計画書のアンチパターン

### 6. 理想論だけの計画書

**症状:**
- 「全員100%アサイン」前提でスケジュール作成
- 休暇、他業務、突発対応を一切考慮しない
- バッファゼロの完璧主義スケジュール

**典型例:**
```
【悪い例】
要件定義フェーズ: 2か月
- 担当者A: 100%アサイン
- 担当者B: 100%アサイン
（実際は他案件兼務、有給休暇、急な会議対応あり）

バッファ: なし
（「計画通り進めれば大丈夫」という楽観）
```

**なぜダメか:**
- 初日から遅延が確定している
- 実績が計画から乖離し、進捗管理が形骸化
- チームが疲弊し、品質が低下

**対策:**
- 稼働率は70-80%で計画（残り20-30%は他業務・調整・バッファ）
- クリティカルパスには15-20%のバッファを明示的に設定
- 「楽観・標準・悲観」の3点見積もりを採用

---

### 7. WBS作成の「とりあえず細分化」

**症状:**
- 意味のない細分化で項目数だけ増やす
- 工数見積もりができないレベルまで細かくする
- 逆に、粗すぎて進捗管理できない

**典型例:**
```
【悪い例 - 細かすぎ】
1.1.1 要件ヒアリング準備
1.1.2 要件ヒアリング資料作成
1.1.3 要件ヒアリング資料レビュー
1.1.4 要件ヒアリング資料修正
（以下、200項目続く...）

【悪い例 - 粗すぎ】
1. 要件定義: 3か月
2. 設計: 4か月
3. 開発: 6か月
（これだけ）
```

**なぜダメか:**
- 細かすぎ: メンテナンスコストが高く、誰も更新しない
- 粗すぎ: 進捗の可視化ができず、問題の早期発見ができない

**対策:**
- WBSの最下位レベルは「1-2週間で完了する作業単位」が目安
- 「8/80ルール」: 1タスクは8時間以上、80時間以下
- 成果物ベースでWBSを構成（作業ベースは下位レベルで）

---

### 8. コピペリスク一覧

**症状:**
- PMBOKの教科書からリスクカテゴリーをコピー
- 一般論すぎて、このプロジェクト固有のリスクが見えない
- 対応策が「注意する」「管理する」など抽象的

**典型例:**
```
【悪い例】
リスク: 「技術的リスク」
対応策: 「技術的な検証を行う」

リスク: 「人的リスク」
対応策: 「適切な人員配置を行う」

リスク: 「スケジュールリスク」
対応策: 「進捗を管理する」
```

**なぜダメか:**
- 具体的な対応アクションにつながらない
- 本当に危険なリスクが埋もれる
- リスク管理をやっているフリになる

**対策:**
```
【良い例】
リスク: 「ベンダーXの主担当Yさんが唯一の技術保有者で、退職・異動リスクあり（発生確率30%、影響度大）」
対応策: 
- 即座実施: ナレッジ共有会を月2回開催、ドキュメント化
- トリガー設定: 退職の兆候があれば即座にベンダーと代替要員確保の交渉
- 回避策: 可能であれば技術を2名体制に変更（今月中に交渉）
```

---

### 9. 「変更管理? 何それ?」問題

**症状:**
- 変更管理プロセスを定義していない
- 口頭での変更依頼を無制限に受け入れる
- 影響分析なしで「できます!」と即答

**典型例:**
```
【悪い例のフロー】
1. ステークホルダー「ここ、こうしてほしいんだけど」（廊下での立ち話）
2. PM「わかりました!」（メモも取らず）
3. 開発チーム「聞いてないです」（後日）
4. PM「え、伝えてなかったっけ?」
5. スケジュール・予算への影響が後から判明し大混乱
```

**なぜダメか:**
- スコープクリープが制御不能に
- 「言った/言わない」が頻発
- 累積した小変更で予算・期間が破綻

**対策:**
```
【良い例のプロセス】
1. 変更要求票の提出を必須化（口頭NG）
2. 影響分析（スコープ・スケジュール・コスト・リスク）
3. 変更審議会（週次）で優先度判定
4. 承認後に実施、却下・保留も選択肢
5. 変更履歴を一元管理
```

---

### 10. 「計画書は作ったら終わり」症候群

**症状:**
- 計画書承認後、一度も開かれない
- 実態と乖離しても更新しない
- 「計画書は形式的なもの」という認識

**典型例:**
```
【悪い例】
- 計画書作成: 2025年1月
- 最終更新: 2025年1月（作成時のまま）
- 現在: 2025年12月
- 状況: スコープ・体制・スケジュールすべて変わったが、計画書は旧版のまま
```

**なぜダメか:**
- 新規参画者が混乱する
- 引継ぎ時に使えない
- 監査・レビューで信頼性ゼロ

**対策:**
- 「生きた文書(Living Document)」として位置づけ
- 版管理を明確化（Major変更時は版番上げ、Minor変更は枝番）
- 月次レビュー会議で計画書の鮮度をチェック
- 大きな変更時は再承認フローへ

## ゲート審査のアンチパターン

### 11. 形式的審査会

**症状:**
- 審査者が資料を事前に読んでいない
- 当日、資料を初めて見て「よくわからないけど承認」
- 実質10分で終わる形だけの審査

**典型例:**
```
【悪い例の審査会】
司会: 「では、プロジェクトXの審査を始めます」
審査者A: 「（資料パラパラ）...まあ、大丈夫そうですね」
審査者B: 「特に意見なし」
司会: 「では承認ということで。次の議題へ」
（所要時間: 5分）
```

**なぜダメか:**
- 重大な問題が見逃される
- ガバナンスが機能していない
- 失敗プロジェクトが量産される

**対策:**
- 資料を3営業日前に配布し、事前レビューを義務化
- チェックリストを使った採点を事前提出
- 審査会では「質問・懸念事項」の議論に集中
- 最低30分は確保、必要なら継続審議

---

### 12. 「全件承認」の儀式

**症状:**
- 審査で「否認」「条件付き承認」が一度も出ない
- どんな計画でも必ず承認される
- 「審査=承認セレモニー」という文化

**典型例:**
```
【悪い例の実績】
過去3年間の審査実績:
- 審査案件: 50件
- 承認: 50件
- 条件付き承認: 0件
- 否認: 0件
- 否認率: 0%（実質的にゲートが機能していない）
```

**なぜダメか:**
- 審査の意味がない
- 準備が甘くても通るため、品質が低下
- 本当に問題のあるプロジェクトが進行してしまう

**対策:**
- 「適切に審査すれば、一定割合は条件付き/否認が出るはず」という認識
- 目安: 条件付き承認20-30%、否認5-10%程度は正常
- 否認・差し戻しは「プロジェクトの質を上げるため」とポジティブに捉える
- 審査者の評価に「適切に否認できたか」を含める

---

### 13. マイルストーン混同問題

**症状:**
- MS1とMS2の区別が曖昧
- 企画段階で本格開発を始めてしまう
- 逆に、実行フェーズなのに延々と計画を作り続ける

**典型例:**
```
【悪い例 - MS1で暴走】
MS1承認: 「企画・調査枠として500万円承認」
→ PM、いきなり3000万円のライセンス発注
→ 「えっ、MS2承認前ですよね...?」

【悪い例 - MS2でも動かない】
MS2承認: 「本格実行OK、予算5000万円コミット」
→ PM「まだリスク分析が不十分なので、あと3か月計画を練ります」
→ 「いつ始めるんですか...?」
```

**なぜダメか:**
- MS1暴走: 承認されていない活動で予算浪費、後で問題化
- MS2停滞: 予算は確保されているのに成果が出ない

**対策:**
- マイルストーンごとの「実施可能活動」「制約事項」を明文化（本資料に記載済み）
- MS1→MS2移行時に「移行チェックリスト」で確認
- PMOが四半期ごとに全案件のフェーズ適合性を監査

## 組織・文化的アンチパターン

### 14. 「ウチは小さいから憲章不要」論

**症状:**
- 「大企業のやり方だから」と導入を拒否
- 口頭合意だけでプロジェクト開始
- 案件規模問わず、全部同じ軽い扱い

**なぜダメか:**
- 小規模でも、曖昧さは同じように問題を起こす
- 後で「言った/言わない」は規模に関係なく発生
- むしろ小規模組織ほど、属人化防止のため文書化が重要

**対策:**
- 小規模組織向けの「ライト版プロジェクト憲章」（A4で1枚）を用意
- 必須項目を5つに絞る: 目的・成果目標・期間・予算・責任者
- 「15分で作れる憲章テンプレート」を整備

---

### 15. PMO不在の放任主義

**症状:**
- 各部門が好き勝手にプロジェクトを乱立
- 標準プロセスがなく、属人的
- リソース競合が発生しても調整する仕組みがない

**典型例:**
```
【悪い例】
- 営業部: 「CRM刷新プロジェクト」を独自に開始
- 製造部: 「生産管理システム更新」を独自に開始
- IT部: 「基盤統合プロジェクト」を独自に開始
→ 3つとも同じベンダーに発注、リソース奪い合い
→ 全部が遅延・品質低下
→ 誰も全体最適を見ていない
```

**なぜダメか:**
- 重複投資・リソース浪費
- 優先順位がつかず、全部が中途半端
- ナレッジ共有されず、同じ失敗を繰り返す

**対策:**
- 小規模でもPMO機能を設置（兼務でも可）
- 全プロジェクトの可視化（ポートフォリオ管理）
- 四半期ごとの優先順位付け会議
- 標準テンプレート・チェックリストの整備と展開

## よくある言い訳とその反論

| 言い訳 | 実態 | 反論 |
|-------|------|------|
| 「時間がないから憲章は後で」 | 後で作ることは永遠にない | 憲章作成は2-4時間。後の手戻りは数週間 |
| 「小規模だから計画書は不要」 | 小規模でも混乱は起きる | A4で2枚の簡易版でもよいので作る |
| 「ウチの文化に合わない」 | 「合わない」=「やりたくない」 | では今の失敗率をどう改善する? |
| 「形式的なので意味ない」 | 形式的にしているのは運用側 | 本気で審査すれば機能する |
| 「前例がないから」 | 誰かが始めなければ永遠にない | パイロットプロジェクトで試行 |
| 「承認に時間かかりすぎ」 | プロセスが複雑すぎる | 簡略版を作り、段階的に改善 |

<br>


## まとめ: アンチパターンを避けるための10の原則

1. **曖昧さを残さない**: 「後で決める」は禁句
2. **測定可能にする**: 数値で表せない目標は目標ではない
3. **Out-of-Scopeを明記する**: やらないことを決める勇気
4. **スポンサーをコミットさせる**: 丸投げを許さない
5. **現実的に計画する**: 理想論・楽観論を排除
6. **変更を管理する**: 口頭変更は即座に拒否
7. **生きた文書にする**: 作りっぱなしは死んだ文書
8. **本気で審査する**: セレモニーではなく品質ゲート
9. **段階的に権限を与える**: MS1とMS2を明確に区別
10. **小さく始めて改善する**: 完璧を目指さず、まず始める

---

**重要な心構え:**

アンチパターンの多くは「面倒だから省略」「形式的でいい」という意識から生まれます。しかし、プロジェクト憲章・計画書の目的は「官僚的な書類作成」ではなく、**「プロジェクトの成功確率を上げる」**ことです。

適切に運用すれば:
- 手戻りが減る（工数削減）
- 意思決定が速くなる（権限明確化）
- チームの納得感が上がる（透明性）
- 失敗の早期発見ができる（ゲート機能）

**「面倒」を避けた結果、何倍もの「混乱」が待っています。**
