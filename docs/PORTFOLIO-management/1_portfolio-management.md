# IT投資ポートフォリオ概要

## IT投資ポートフォリオ管理の全体フロー

```mermaid
sequenceDiagram
    participant ビジネス部門
    participant ファイナンス部門
    participant IT投資委員会
    participant IT部門
    participant ポートフォリオ管理
    participant プロジェクトチーム

    Note over ビジネス部門,ファイナンス部門: 【ステップ0】予算提出と答申対象案件リスト作成
    ビジネス部門->>ファイナンス部門: 予算提出<br/>(IT投資案件の予算要求)
    ファイナンス部門->>ファイナンス部門: IT投資委員会へ答申対象の<br/>案件リストを作成
    ファイナンス部門->>IT投資委員会: 答申対象案件リスト共有
    IT投資委員会->>ポートフォリオ管理: 答申対象案件リスト共有
    ポートフォリオ管理->>ポートフォリオ管理: 既存案件と新規案件の<br/>分類（既存/新規/不明）
    ポートフォリオ管理->>IT投資委員会: 案件分類結果を共有
    IT投資委員会->>ビジネス部門: 答申対象案件リスト<br/>(案件分類つき)共有<br/>案件概要書作成指示
    ビジネス部門->>ビジネス部門: 答申対象案件リスト確認<br/>(案件分類・ポートフォリオ確認)
    ビジネス部門->>ビジネス部門: 今回の答申対象を決定
    ビジネス部門->>IT投資委員会: 答申対象決定を通知
    IT投資委員会->>IT投資委員会: 答申対象案件リスト確定
    IT投資委員会->>ファイナンス部門: 確定した答申対象案件リスト共有

    Note over ビジネス部門: 【ステップ1】案件概要書の作成
    ビジネス部門->>ビジネス部門: 案件概要書作成<br/>(目的・期待効果・予算等)
    ビジネス部門->>ビジネス部門: 案件タイプ自己申告<br/>(Type A/B/C/D)
    ビジネス部門->>IT投資委員会: 案件概要書提出

    Note over IT投資委員会: 【ステップ2】IT評価書の作成
    IT投資委員会->>IT投資委員会: 案件タイプ判定
    IT投資委員会->>IT部門: IT評価書作成<br/>(技術的実現性・リソース評価)
    IT部門->>IT投資委員会: 案件概要書+IT評価書提出

    Note over IT投資委員会: 【ステップ3】IT投資委員会での審査
    IT投資委員会->>ビジネス部門: 18の質問項目に基づく審査
    ビジネス部門->>IT投資委員会: 質問への回答
    IT投資委員会->>IT投資委員会: 評価・採択判断<br/>(採択/不採択/保留)
    
    alt 採択
        IT投資委員会->>ポートフォリオ管理: 採択通知
    else 不採択
        IT投資委員会->>ビジネス部門: 不採択通知
    else 保留
        IT投資委員会->>ビジネス部門: 保留通知<br/>(再審査時期指定)
    end

    Note over ポートフォリオ管理: 【ステップ4】ポートフォリオへの登録
    ポートフォリオ管理->>ポートフォリオ管理: 採択案件をポートフォリオ登録
    ポートフォリオ管理->>ビジネス部門: ポートフォリオ登録完了<br>ポートフォリオレビュー日程通知

    Note over ポートフォリオ管理: 【ステップ5】定期レビューの実施
    loop 月次レビュー
        ポートフォリオ管理->>ビジネス部門: 進捗状況報告依頼
        ビジネス部門->>ポートフォリオ管理: 進捗状況報告
        ポートフォリオ管理->>IT投資委員会: 月次レビュー実施<br/>(進捗確認・課題対応)
    end
    
    loop 四半期レビュー
        ポートフォリオ管理->>IT投資委員会: 四半期レビュー実施<br/>(優先順位見直し・予算再配分)
        alt 優先順位・予算見直し
            IT投資委員会->>ポートフォリオ管理: 優先順位・予算変更指示
            ポートフォリオ管理->>ポートフォリオ管理: ポートフォリオ更新
        end
    end

    Note over ビジネス部門,プロジェクトチーム: 【ステップ6】プロジェクト化
    ビジネス部門->>ビジネス部門: プロジェクト憲章作成
    ビジネス部門->>ポートフォリオ管理: MS1承認申請<br/>(プロジェクト憲章)
    ポートフォリオ管理->>ビジネス部門: MS1承認
    
    Note over ビジネス部門: 企画・計画策定フェーズ<br/>(予算: 全体の数%〜10%)
    ビジネス部門->>ビジネス部門: プロジェクト実行計画作成<br/>(WBS・スケジュール・リソース計画)
    ビジネス部門->>ポートフォリオ管理: MS2承認申請<br/>(プロジェクト実行計画)
    ポートフォリオ管理->>ビジネス部門: MS2承認
    
    Note over プロジェクトチーム: 実行フェーズ<br/>(予算: 全体予算を正式コミット)
    ビジネス部門->>プロジェクトチーム: プロジェクト実行開始
    プロジェクトチーム->>プロジェクトチーム: プロジェクト実行
    プロジェクトチーム->>ポートフォリオ管理: 進捗報告(定期)
```

## 新規提案から案件化まで

```
【提案フェーズ】
ビジネス部門
  ├─ 案件概要書作成（全案件共通）
  └─ IT部門へ提出
       ↓
IT部門
  ├─ 案件タイプ判定（A/B/C/D）
  ├─ 該当するIT評価書作成
  │   ├─ Type A: ベンダー提案型
  │   ├─ Type B: 製品導入型
  │   ├─ Type C: 既存改修型
  │   └─ Type D: 新規開発型
  └─ IT投資委員会へ提出
       ↓
【IT投資委員会】
  ├─ 案件概要書 + IT評価書をセットで審査
  ├─ 採択/不採択判断
  ├─ 優先順位付け
  └─ 予算配賦決定
       ↓
【IT投資ポートフォリオ】
  採択案件のリスト
  ├─ 優先順位
  ├─ 予算配賦額
  ├─ 想定スケジュール
  ├─ リソース配分計画
  └─ 依存関係マップ
       ↓
【プロジェクト化】
  各案件について
  ├─ プロジェクト憲章作成（MS1承認）
  ├─ プロジェクト実行計画作成（MS2承認）
  └─ プロジェクト実行
```

### IT評価の案件タイプ

| タイプ | 説明 | 評価対象の有無 | 評価の焦点 |
|--------|------|---------------|-----------|
| **Type A: ベンダー提案型** | 既存ITベンダーから具体的な提案を受けている | ◎ 提案書あり | 提案内容の妥当性評価 |
| **Type B: 製品導入型** | SaaS/パッケージ等、製品が存在している | ◎ 製品仕様あり | 製品適合性評価 |
| **Type C: 既存改修型** | 既存システムの改修・機能追加 | △ 既存システムあり | 改修影響範囲評価 |
| **Type D: 新規開発型** | ゼロから開発が必要 | × 評価対象なし | 技術選定・実現性評価 |

## ポートフォリオ段階での文書

### 文書作成の課題

- **ポートフォリオレベル**：案件の採択・優先順位付け・予算配賦
- **プロジェクトレベル**：プロジェクト憲章（MS1）→ 計画書（MS2）

**ギャップ**：
- ポートフォリオで「採択判断」するには、何らかの**案件情報**が必要
- しかし、プロジェクト憲章はポートフォリオ承認**後**に作成するのか？

### 案件概要書・IT評価書の作成

```
【ビジネス部門】
案件概要書（1-2ページ）
  ├─ 目的・期待効果
  ├─ 概算予算・期間
  ├─ ビジネスインパクト
  └─ 概略リスク
    ↓
【IT部門】
IT評価書
  ├─ 技術的実現性（インフラ・セキュリティ・運用負荷・技術負債・他案件との依存関係）
  └─ リソース制約（保有リソース・リソース競合）
    ↓
【IT投資委員会】
両文書をセットで審査
採択・優先順位付け
    ↓
【プロジェクト段階】
プロジェクト憲章（MS1承認用）
  └─ より詳細化した内容
```

**メリット:**
- ポートフォリオ判断に必要な情報のみに絞れる
- 不採択案件への工数を最小化

**デメリット:**
- 文書が重複する可能性
- 案件概要書→プロジェクト憲章の整合性管理が必要

### RACIの再定義

| 役割 | ビジネス部門 | IT部門 | デジタル部門 | IT投資委員会 |
|------|-------------|--------|-------------|-------------|
| **案件概要書作成** | R/A | C | C | - |
| **IT評価書作成** | I | R/A | C | - |
| **ポートフォリオ採択判断** | C | C | C | **R/A** |
| **プロジェクト憲章ドラフト作成** | R/A | C | C | I |
| **優先順位確定・予算配賦** | C | C | C | **R/A** |
| **憲章正式版承認（MS1）** | **R/A** | C | C | I |

## IT投資ポートフォリオの管理項目

### 1. 採択案件の管理

**記録事項:**
- 採択日時・採択理由
- 不採択案件の記録（理由・再提案条件）
- 保留案件の記録（保留理由・再審査時期）

### 2. 優先順位付けの基準

**評価軸（例）:**
| 評価軸 | 配点 | 判断基準 |
|--------|------|---------|
| ビジネスインパクト | 30点 | 売上・コスト削減効果 |
| 戦略適合性 | 25点 | 経営戦略・DX戦略との整合性 |
| 緊急性 | 20点 | 法対応、システム老朽化等 |
| 実現可能性 | 15点 | IT評価書での判定 |
| リソース制約 | 10点 | 人材・インフラの確保可能性 |

### 3. 予算配賦の管理

**予算枠の分類:**
```
全社IT投資予算: 10億円/年
  ├─ 戦略的投資枠: 6億円
  │   ├─ DX推進案件: 3.5億円
  │   ├─ 業務効率化案件: 2.0億円
  │   └─ 新規事業案件: 0.5億円
  │
  ├─ 維持・延命投資枠: 3億円
  │   ├─ システム更改: 2.0億円
  │   └─ 保守・改修: 1.0億円
  │
  └─ 予備枠: 1億円
      └─ 緊急対応・機会対応
```

### 4. リソース配分計画

**IT人材のアサイン計画:**
```
2025年度 IT人材配分（延べ人月）
  ├─ SAP専門家: 36人月 → 3案件に配分
  ├─ クラウドエンジニア: 48人月 → 4案件に配分
  ├─ データサイエンティスト: 24人月 → 2案件に配分
  └─ PMO: 60人月 → 全案件支援
```

## IT投資ポートフォリオの定期レビュー

### レビューサイクル

| レビュー種別 | 頻度 | 目的 | 参加者 |
|-------------|------|------|--------|
| **月次レビュー** | 毎月 | 進捗確認、課題対応 | IT投資委員会事務局 |
| **四半期レビュー** | 3ヶ月毎 | 優先順位見直し、予算再配分 | IT投資委員会 |
| **年次レビュー** | 年1回 | 新年度ポートフォリオ策定 | 経営会議・IT投資委員会 |

### レビューで確認する事項

1. **進捗状況**
   - 予定通り進行中の案件
   - 遅延している案件とその影響
   - 完了した案件の成果

2. **リソース状況**
   - IT人材の稼働状況
   - リソース競合の発生
   - 外部リソースの活用状況

3. **予算執行状況**
   - 予算消化率
   - 超過見込み案件
   - 未消化予算の再配分

4. **新規案件の追加**
   - 緊急性の高い案件の割り込み
   - 予備枠からの予算配賦

5. **案件の中止・延期判断**
   - 期待効果が見込めなくなった案件
   - リソース制約により継続困難な案件

## ポートフォリオからプロジェクトへの移行

### 移行タイミング

```
IT投資ポートフォリオ承認
    ↓
【準備期間】
  ├─ プロジェクト憲章ドラフト作成
  ├─ PMアサイン
  └─ 初期体制構築
    ↓
【MS1承認】
  プロジェクト憲章正式承認
    ↓
【企画・計画策定フェーズ】
  ├─ 詳細要件定義
  ├─ ベンダー選定
  └─ プロジェクト実行計画作成
    ↓
【MS2承認】
  プロジェクト実行計画承認
    ↓
【実行フェーズ】
  プロジェクト実行
```
