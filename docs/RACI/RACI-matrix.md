# 組織変革のためのRACIマトリクス検証ガイド

## 1. RACIマトリクスとは

### 1.1 定義
RACIマトリクスは、プロジェクトやタスクにおける**役割と責任を明確化するための管理ツール**です。縦軸にタスク・成果物、横軸にチームメンバーや部門を配置し、交差するセルに役割を記入します。

### 1.2 RACIの意味

| 記号 | 役割 | 説明 | コミュニケーション |
|------|------|------|-------------------|
| **R** | **Responsible（実行責任者）** | 実際に作業を行う人。タスクの実行者 | - |
| **A** | **Accountable（説明責任者）** | 最終的な責任を持つ人。承認権限を持つ。**1タスク1人** | - |
| **C** | **Consulted（協議先）** | 作業前や作業中に意見を求められる人 | **双方向** |
| **I** | **Informed（報告先）** | 進捗や結果を知らせる必要がある人 | **一方向** |

### 1.3 RACIマトリクスの目的

- **役割分担の明確化**: 「誰が何をするか」が一目でわかる
- **責任の所在を明確化**: 「誰が最終決定するか」が明確で、責任が曖昧にならない
- **重複と抜け漏れの防止**: 作業の重複や、誰も担当しないタスクを防ぐ
- **コミュニケーションの最適化**: 誰に相談し、誰に報告すべきかが明確
- **プロジェクトの可視化**: チーム全体でプロジェクトの全体像を共有

### 1.4 重要な原則

#### 原則1: 1タスク1人のAccountable（A）
- 各タスクには必ずAが**1人だけ**存在する
- 複数のAがいると責任が分散し、意思決定が遅れる
- Aがいないタスクは承認者不在で進められない

#### 原則2: 必ずResponsible（R）が存在
- 全てのタスクに最低1人のRが必要
- Rがいないタスクは実行されない

#### 原則3: バランスの取れた分散
- 特定の人にR/Aが集中しすぎると、ボトルネックになる
- C（協議先）やI（報告先）が多すぎると、情報過多で非効率

## 2. RACIマトリクス検証のテクニックとアプローチ

RACIマトリクスが実際に機能するかを検証するため、以下のテクニックを段階的に適用します。

### 2.1 検証テクニック一覧

| No | テクニック | 目的 | 推奨タイミング |
|----|-----------|------|--------------|
| 1 | RACI検証チェックリスト | マトリクスの形式的整合性確認 | 作成直後 |
| 2 | ステークホルダインタビュー | 実行可能性と現場感覚の確認 | 初期検証 |
| 3 | ワークショップ | 関係者全員での集合的検証 | 合意形成 |
| 4 | ウォークスルー | 実際の業務フローでの追体験 | 詳細検証 |
| 5 | パイロットテスト | 小規模な実運用での検証 | 最終検証 |

### 2.2 推奨アプローチ（段階的検証）

```
第1段階: RACI検証チェックリスト
  ↓ 形式的な問題を修正
  
第2段階: ステークホルダインタビュー
  ↓ 個別の懸念事項を収集
  
第3段階: ワークショップ形式のレビュー
  ↓ 関係者全員で合意形成
  
第4段階: ウォークスルー/シミュレーション
  ↓ 実際の業務で検証
  
第5段階: パイロットテスト
  ↓ 実運用でブラッシュアップ
  
完成・本格展開
```

### 2.3 各検証テクニックの詳細

#### 2.3.1 RACI検証チェックリスト

**目的**: マトリクスの論理的整合性を確認

**チェック項目**:
- [ ] 各タスクに必ずAが1人だけ存在するか
- [ ] 各タスクに最低1人のRが存在するか
- [ ] Aが多すぎて承認が遅延しないか（全タスクの30%以下が目安）
- [ ] CやIが多すぎて情報過多にならないか
- [ ] 特定の人にR/Aが集中しすぎていないか（全タスクの50%以下が目安）
- [ ] 現実的な作業負荷になっているか

#### 2.3.2 ステークホルダインタビュー

**目的**: 各役割を担う人の実行可能性と懸念を確認

**インタビュー項目**:
1. この役割分担は実行可能ですか？（時間、スキル、リソース面）
2. 責任範囲は明確ですか？過不足はありませんか？
3. 他部門との連携はイメージできますか？
4. 現状の役割とのギャップはどの程度ですか？
5. この体制で問題が起きそうなポイントはどこですか？

**収集すべき情報**:
- 実行可能性の評価（5段階など）
- 必要なサポートやリソース
- リスクや懸念事項
- 改善提案

#### 2.3.3 ワークショップ形式のレビュー

**目的**: 関係者全員を集めて集合的に検証し、合意形成

**進め方**:
1. ファシリテーターを立てる
2. フェーズごとに参加者で議論
3. 付箋やホワイトボードで課題を可視化
4. その場で修正案を検討
5. 全員の合意を得る

**ワークショップの利点**:
- 認識のズレがその場で解消される
- 当事者意識が高まる
- 部門間の壁を超えた議論ができる

#### 2.3.4 ウォークスルー

**目的**: 実際の業務シナリオに沿って、時系列で役割を追体験

**実施方法**:
1. 具体的な業務シナリオを用意（過去案件または想定案件）
2. 各フェーズのタスクを時系列で確認
3. 各ステップで「誰が実際に動くか」「誰の承認が必要か」を検証
4. ボトルネック、役割の重複・空白を発見

**確認ポイント**:
- タスクの前後関係は整合しているか
- 承認（A）が遅延する可能性はないか
- 協議（C）が必要な意思決定ポイントは明確か
- 情報の流れ（I）は適切なタイミングか

#### 2.3.5 パイロットテスト

**目的**: 小規模な実プロジェクトで試験的に適用

**実施内容**:
1. 限定的な範囲でRACIを実運用
2. 定期的に振り返り会議を実施（週次または隔週）
3. 問題点をリアルタイムで記録
4. 実績データに基づき改善
5. 改善版RACIを本格展開

**測定すべき指標**:
- タスク完了率
- 承認の所要時間
- やり直し・手戻りの発生率
- 関係者の満足度

## 3. タスクをビジネスプロセスにする目的

### 3.1 タスクとビジネスプロセスの違い

| 観点 | タスク | ビジネスプロセス |
|------|--------|----------------|
| 粒度 | 個別の作業単位 | タスクの連鎖・流れ |
| 視点 | 点（What） | 線（How） |
| 時間軸 | 単発 | 時系列・フロー |
| 関係性 | 独立 | 前後依存関係あり |
| 例 | 「承認する」 | 「申請→確認→承認→通知」 |

### 3.2 ビジネスプロセス化の目的

#### 目的1: 全体像の可視化
**なぜ必要か**: 
- タスクだけでは「木を見て森を見ず」の状態になる
- プロセス全体を俯瞰することで、業務の流れが理解できる

**効果**:
- どのタスクがどこに位置するか明確になる
- インプットとアウトプットの関係が見える
- ボトルネックの発見が容易になる

#### 目的2: 依存関係の明確化
**なぜ必要か**:
- タスクは単独では完結せず、前工程の成果物が必要
- 前後関係を無視すると、情報不足や手戻りが発生

**効果**:
- 「Aのタスクが終わらないとBができない」という制約が明確
- クリティカルパスが特定できる
- 並行実施可能なタスクが判別できる

#### 目的3: 情報フローの設計
**なぜ必要か**:
- RACIのC（協議）やI（報告）は、タイミングが重要
- プロセスの流れに沿って、適切なタイミングで情報共有が必要

**効果**:
- 「誰に、いつ、何を伝えるか」が明確になる
- 情報の重複や漏れを防げる
- コミュニケーションコストを最適化できる

#### 目的4: 標準化と改善の基盤
**なぜ必要か**:
- 組織変革では、業務の標準化が前提
- プロセスとして可視化しないと、標準化も改善もできない

**効果**:
- 業務のベストプラクティスが明文化される
- 属人化が解消される
- 継続的な改善（PDCA）が可能になる

### 3.3 ビジネスプロセス化のアプローチ

```
タスクリスト
  ↓ グルーピング
フェーズ分け
  ↓ 時系列整理
業務フロー作成
  ↓ 依存関係追加
ビジネスプロセスモデル
  ↓ RACI割り当て
実行可能なプロセス
```

## 4. ビジネスプロセスを業務シナリオにしてウォークスルーする目的

### 4.1 業務シナリオとは

**定義**: 
ビジネスプロセスを、**具体的な人物（ペルソナ）が実際に業務を行う物語**として記述したもの。

**業務シナリオの要素**:
- 登場人物（ペルソナ）: 役割、経験、スキルレベル
- 状況設定: いつ、どこで、何のために
- 具体的な行動: 何を見て、何を考え、何をするか
- 数値・データ: 処理件数、所要時間、頻度など
- 障害・例外: 想定されるトラブルとその対処

### 4.2 業務シナリオ化の目的

#### 目的1: 抽象から具体への変換
**なぜ必要か**:
- プロセス図は抽象的で、実際の業務がイメージしにくい
- 「承認する」という記述だけでは、実際の動きが見えない

**効果**:
- 「山田部長がスマホで通知を受け取り、移動中の電車内で内容を確認し、承認ボタンを押す」という具体性
- 関係者全員が同じイメージを持てる
- 現場の実態に即しているか判断できる

#### 目的2: 実行可能性の検証
**なぜ必要か**:
- プロセス図上は問題なくても、実際には実行不可能な場合がある
- 「この人にこの判断ができるか」「この情報は入手可能か」を確認する必要

**効果**:
- シナリオを追体験することで、現実性を検証できる
- 「この段階で必要な情報がない」などの問題を発見
- RACIの実行可能性を具体的にチェックできる

#### 目的3: 例外・エッジケースの発見
**なぜ必要か**:
- 通常フローだけでは、トラブル時の対応が不明確
- 現実の業務には、常に例外が発生する

**効果**:
- 「承認者が不在の場合は？」「データが不正な場合は？」などを想定
- エスカレーションルールやバックアッププランを設計
- より堅牢なプロセスになる

#### 目的4: ステークホルダの当事者意識向上
**なぜ必要か**:
- 抽象的なプロセスでは、自分事として捉えにくい
- シナリオで具体的に描写することで、リアリティが生まれる

**効果**:
- 「自分がこの役割を担ったら…」と想像できる
- 問題点や改善点を積極的に指摘するようになる
- プロジェクトへのコミットメントが高まる

#### 目的5: RACIマトリクスの実証
**なぜ必要か**:
- RACIはルール上正しくても、実際の業務で機能するとは限らない
- シナリオで追体験することで、初めて実効性が確認できる

**効果**:
- 「このタイミングでこの人に協議（C）が必要」などの妥当性を検証
- 「この人がAで本当に承認できるか」を具体的に確認
- 役割の過不足、情報フローの問題を発見

### 4.3 ウォークスルーの実施方法

#### ステップ1: シナリオ準備
1. 代表的な業務ケースを選定（頻度が高い、重要度が高い）
2. 登場人物（ペルソナ）を設定
3. 具体的な数値やデータを盛り込む
4. As-Is（現状）とTo-Be（改善後）の両方を用意

#### ステップ2: ウォークスルー実施
1. 参加者全員でシナリオを読み合わせ
2. 各タスクで「誰が」「何を」「どのように」行うか確認
3. RACIマトリクスと照らし合わせ
4. 問題点や疑問点をその場で記録

#### ステップ3: 検証ポイント
- **R（実行責任者）**: この人が本当に実行できるか？必要な情報・権限・スキルはあるか？
- **A（説明責任者）**: この人が承認判断できるか？承認基準は明確か？タイミングは適切か？
- **C（協議先）**: この段階で協議が必要か？協議内容は明確か？
- **I（報告先）**: この人に報告する必要性は？タイミングは適切か？

#### ステップ4: 改善と反映
1. 発見された問題をリスト化
2. 優先順位をつけて対応策を検討
3. RACIマトリクスまたはプロセスを修正
4. 再度ウォークスルーで検証

### 4.4 ウォークスルーで発見できる典型的な問題

| 問題の種類 | 具体例 | 対策 |
|----------|-------|------|
| 情報不足 | 承認時に必要なデータが入手できない | プロセスに情報収集タスクを追加 |
| 権限不足 | Aに指定された人に承認権限がない | Aを適切な権限者に変更 |
| スキル不足 | Rに指定された人にスキルがない | 研修計画の策定、または担当変更 |
| タイミング問題 | 報告（I）が遅すぎて意思決定に間に合わない | 情報共有のタイミングを前倒し |
| ボトルネック | 特定の人（A）に承認が集中して遅延 | 承認権限の委譲、並行処理の検討 |
| 重複作業 | 複数の人が同じ確認作業をしている | 役割を整理し、Rを一本化 |
| 例外対応不明 | エラー時の対応が定義されていない | 例外フローを追加 |

## 5. 実践の流れ

### 5.0 As-Is（第一版）とTo-Be（第二版）の段階的アプローチ

**重要な考え方**: 組織変革では、現行体制を踏襲した上で段階的に改善することが成功の鍵です。

#### 第一版（As-Isモデル）の目的
**なぜ現行モデルから始めるか**:
- 組織変革では、いきなり理想形に移行すると混乱が生じる
- まず現状の役割分担を明確化し、体制移行の基盤を作る
- 現行業務の可視化により、問題点が明確になる
- ステークホルダが「現状を正しく理解する」ことが改善の第一歩

**第一版で達成すること**:
1. 現状の役割分担を正確に可視化する
2. 暗黙知や属人化している部分を明文化する
3. 現行体制でのボトルネックや課題を特定する
4. 組織変革の出発点として合意を得る

#### 第二版（To-Beモデル）の位置づけ
**第一版を試金石として**:
- 第一版の運用経験から、改善点を具体的に抽出
- 「こうあるべき」ではなく「こうすれば改善できる」という実証ベース
- 段階的な移行計画により、リスクを最小化

**第二版で実現すること**:
1. 第一版で発見された課題を解決する役割分担
2. 組織変革による業務効率化を反映した体制
3. 標準化・自動化を前提とした新しいプロセス
4. 継続的改善が可能な柔軟な組織体制

## まとめ

### RACIマトリクスの価値
- プロジェクトにおける役割と責任を明確化し、曖昧さを排除する
- コミュニケーションを最適化し、効率的な業務遂行を実現する

### 段階的アプローチの重要性
- **第一版（As-Isモデル）**: 組織変革の出発点として、現状を正確に可視化
  - 現行体制を踏襲することで、混乱を最小化
  - 暗黙知を明文化し、組織全体で認識を統一
  - 改善の基盤となる課題を明確化
- **第二版（To-Beモデル）**: 第一版の運用経験を基に、実証的に改善
  - 「こうあるべき」ではなく「こうすれば改善できる」という実証ベース
  - 組織変革・標準化を反映した効率的な体制
  - 段階的移行によりリスクを最小化

### 検証の重要性
- RACIは「作って終わり」ではなく、**実際に機能するか検証**が不可欠
- 段階的な検証アプローチで、実効性を高める

### ビジネスプロセス化の意義
- タスクを点で見るのではなく、**線（フロー）で捉える**ことで全体最適を実現
- 依存関係や情報フローが明確になり、標準化とシステム化の基盤になる

### 業務シナリオとウォークスルーの効果
- 抽象的なプロセスを**具体的な物語**にすることで、実行可能性を検証
- 例外ケースや隠れた問題を発見し、より堅牢なプロセスを構築
- ステークホルダの当事者意識を高め、プロジェクトの成功確率を向上

### 組織変革への寄与
RACIマトリクスとその検証プロセスは、**組織変革における業務可視化・標準化の中核**となります。
