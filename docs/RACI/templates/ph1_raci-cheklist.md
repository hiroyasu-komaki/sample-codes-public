# RACI検証テクニック - 成果物テンプレート集

## Phase 1: RACI検証チェックリスト

### テンプレート概要
- **目的**: マトリクスの形式的整合性確認
- **推奨タイミング**: RACIマトリクス作成直後
- **所要時間**: 30分〜1時間
- **実施者**: プロジェクトマネージャー、RACI作成担当者

---

### 1.1 基本チェック項目

#### ✅ 必須要件チェック

| 項目 | チェック内容 | 結果 | 備考 |
|------|------------|------|------|
| 1. Aの一意性 | 各タスクに必ずAが1人だけ存在するか | ☐ OK ☐ NG | 複数A、またはA不在のタスク数: ___ |
| 2. Rの存在 | 各タスクに最低1人のRが存在するか | ☐ OK ☐ NG | R不在のタスク数: ___ |
| 3. R/A重複 | 同一人物が同一タスクでR&Aを兼務していないか | ☐ OK ☐ NG | 兼務タスク数: ___ |
| 4. 空白行確認 | 全てのタスクに何らかの役割が割り当てられているか | ☐ OK ☐ NG | 空白タスク数: ___ |

**判定基準**:
- 全て OK → Phase 2へ進む
- NG が1つでも存在 → 即座に修正が必要

---

### 1.2 バランスチェック

#### 📊 負荷分散の確認

**A（説明責任者）の集中度**

| 担当者名 | A担当タスク数 | 全タスクに占める割合 | 判定 | コメント |
|---------|-------------|------------------|------|---------|
| 例: 山田部長 | 12 | 30% | ⚠️ 高い | 承認遅延リスクあり |
|  |  |  |  |  |
|  |  |  |  |  |
|  |  |  |  |  |

**判定基準**:
- 30%以下: ✅ 適正
- 31-50%: ⚠️ 要注意（ボトルネックの可能性）
- 51%以上: ❌ 危険（承認遅延の高リスク）

---

**R（実行責任者）の集中度**

| 担当者名 | R担当タスク数 | 全タスクに占める割合 | 判定 | コメント |
|---------|-------------|------------------|------|---------|
| 例: 佐藤主任 | 25 | 62% | ❌ 危険 | 作業過多、要分散 |
|  |  |  |  |  |
|  |  |  |  |  |
|  |  |  |  |  |

**判定基準**:
- 50%以下: ✅ 適正
- 51-70%: ⚠️ 要注意（過負荷の可能性）
- 71%以上: ❌ 危険（作業遅延の高リスク）

---

**C（協議先）・I（報告先）の数**

| タスク名 | C の人数 | I の人数 | 合計 | 判定 | コメント |
|---------|---------|---------|------|------|---------|
| 例: 要件定義 | 5 | 8 | 13 | ❌ 過多 | 情報過多、会議時間増加 |
|  |  |  |  |  |  |
|  |  |  |  |  |  |

**判定基準**:
- C+I が 5人以下: ✅ 適正
- C+I が 6-10人: ⚠️ 要検討（情報共有コスト増加）
- C+I が 11人以上: ❌ 過多（会議・メール過多のリスク）

### 1.3 論理整合性チェック

#### 🔍 タスク間の整合性

| 検証項目 | チェック内容 | 結果 | 問題タスク |
|---------|------------|------|----------|
| 前後依存関係 | 前工程のAが後工程のCまたはIに含まれているか | ☐ OK ☐ NG |  |
| 情報フロー | Iに指定された人が適切なタイミングで情報を受け取れるか | ☐ OK ☐ NG |  |
| 協議の妥当性 | Cに指定された人が実際に意思決定に関与すべきか | ☐ OK ☐ NG |  |
| 権限の確認 | Aに指定された人が実際に承認権限を持っているか | ☐ OK ☐ NG |  |
