# IT組織におけるケイパビリティ管理フレームワーク

> 需給ギャップの可視化を中心とした実践的ガイド

## 📖 概要

このドキュメントは、IT組織における**ケイパビリティ管理フレームワーク**について、特に「需給ギャップの可視化」を中心に解説した実践的なガイドです。

IT組織が「どんなスキル・能力が、どれだけ必要で、現状どれだけ足りないのか」を体系的に把握し、対応策を立てるための枠組みを提供します。

---
<br>

## 🎯 核心的なコンセプト

**ケイパビリティ管理の本質**は、感覚的な「人が足りない」という状態から、**「クラウドアーキテクト(L3-4)が7名不足」** といった具体的な把握への転換です。

これにより、以下が実現できます:
- データドリブンな意思決定
- 限られたリソースの最適配分
- 具体的な採用・育成計画の策定
- 予算の根拠ある配分

---
<br>


## 🔑 4つの重要な軸

ケイパビリティは以下の4つの軸で分類されます:

### 1. 技術ケイパビリティ
- クラウド、ネットワーク、セキュリティ
- フロントエンド、バックエンド、モバイル開発
- データエンジニアリング、機械学習、アナリティクス

### 2. ビジネスケイパビリティ
- 業務知識(財務、営業、製造等)
- 業界知識(金融、製造、小売等)
- 規制対応、コンプライアンス

### 3. デリバリーケイパビリティ
- プロジェクト管理(アジャイル、ウォーターフォール)
- 品質管理、テスト
- ベンダー管理、パートナー協業

### 4. リーダーシップケイパビリティ
- アーキテクチャ、ソリューション設計
- イノベーション、デザイン思考
- 変革管理、ピープルマネジメント

## 📊 需給ギャップ分析の流れ

### 需要側(必要な能力)の分析
- プロジェクトポートフォリオから必要人月を算出
- 技術トレンドや戦略方針を反映
- 地域別、時期別の需要を予測

### 供給側(現状の能力)の分析
- 組織のスキルマトリクスで現状を把握
- 自然減(退職、異動)を考慮
- 成長見込み(育成、昇格)を反映

### ギャップの可視化と優先順位付け
**ギャップ = 需要 - 供給** を定量化し、以下の軸で優先順位を設定:
- ビジネス影響度(高/中/低)
- ギャップサイズ(大/小)
- 対応緊急度(3ヶ月/6ヶ月/12ヶ月)

## 💡 実践的なアプローチ

### 段階的詳細化の4フェーズ

このフレームワークの優れた点は、**データが不完全でも始められる段階的アプローチ**を提示していることです:

#### フェーズ1: 量的ギャップ分析(即時実施可能)
- **期間**: 即時
- **精度**: ±30%
- **分析内容**: 総人月レベルの過不足把握
- **意思決定**: 総要員数、予算規模、外部活用の規模感

#### フェーズ2: 粗粒度の質的分析(1-3ヶ月)
- **期間**: 1-3ヶ月
- **精度**: ±20%
- **分析内容**: 5-10カテゴリでの需給分析
- **意思決定**: 採用領域、育成重点、外部活用領域

#### フェーズ3: 中粒度分析(3-6ヶ月)
- **期間**: 3-6ヶ月
- **精度**: ±10%
- **分析内容**: スキル×レベルでの詳細ギャップ
- **意思決定**: 個別採用ターゲット、個人別育成計画

#### フェーズ4: 詳細分析(6-12ヶ月)
- **期間**: 6-12ヶ月以降
- **精度**: ±5%
- **分析内容**: 多次元での最適化分析
- **意思決定**: 予測的人材計画、継続的改善

<br>

### 重要な洞察

> **精度30%でも意思決定の80%は可能**

完璧なデータを待つより、不完全でも行動を開始し、段階的に精度を上げることが重要です。

## 🛠️ ギャップ解消の4戦略(4B戦略)

各ギャップに応じて、以下の戦略を最適に組み合わせます:

### 1. Build(育成)
- **適用場面**: 長期的なコア能力、組織固有知識
- **期間**: 6ヶ月～2年
- **手法**: 構造化育成プログラム、メンタリング、ジョブローテーション
- **例**: 既存エンジニアをクラウドエンジニアに育成(9-12ヶ月)

### 2. Buy(採用)
- **適用場面**: 即戦力が必要、内部育成困難な高度専門性
- **期間**: 3-6ヶ月
- **手法**: 戦略的採用計画、ダイバーシティ採用、リファラル制度
- **例**: クラウドアーキテクト(L4)を市場から2名採用

### 3. Borrow(外部活用)
- **適用場面**: 一時的需要、実験的取組み、ピーク時補完
- **期間**: 3-24ヶ月(形態により変動)
- **形態**: コンサルタント、フリーランス、SES/派遣、オフショア
- **例**: クラウド移行コンサルタントを6ヶ月活用

### 4. Bot(自動化)
- **適用場面**: 反復的タスク、標準化可能作業
- **期間**: 継続的
- **手法**: RPA、AI/ML、Low-code/No-code、DevOps/IaC
- **例**: テスト自動化により需要を30%削減

## 📈 継続的改善サイクル

### KPIによるモニタリング

主要なKPI例:
- **充足率**: 全体的なケイパビリティの充足度(目標: 85%以上)
- **クリティカルギャップ数**: 最優先対応が必要なギャップ(目標: 3件以下)
- **育成完了率**: 育成プログラムの完遂率(目標: 80%以上)
- **採用充足率**: 計画に対する採用実績(目標: 80%以上)
- **リテンション率**: 重要人材の定着率(目標: 95%以上)

### 四半期レビューサイクル

```
Q1実行 → Q1末レビュー → Q2調整・実行 → Q2末レビュー 
→ Q3調整・実行 → Q3末レビュー → Q4調整・実行 
→ 年度末総括 → 次年度計画
```

計画を立てて終わりではなく、定期的にレビューして調整する**生きた仕組み**として運用します。

## ✨ フレームワークの特徴

このフレームワークの最大の特徴は、理論だけでなく**実務での制約を前提とした現実的なアプローチ**を提示している点です:

- ✅ データ不足でも開始可能
- ✅ 予算制限を考慮した優先順位付け
- ✅ 段階的な詳細化による実行性の高さ
- ✅ 4B戦略による柔軟な対応
- ✅ 継続的改善による持続可能性

## 🚀 次のステップ

1. **現状のベースライン把握**: 可能な範囲でデータ収集
2. **需要予測**: プロジェクトポートフォリオからの粗算出
3. **量的ギャップ分析**: 総人月レベルでの過不足把握
4. **対応策の立案**: 優先順位に基づいたアクションプラン
5. **予算化**: 必要リソースの予算申請
6. **実行とモニタリング**: KPIに基づいた進捗管理
