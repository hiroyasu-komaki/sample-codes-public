# MSR（記録管理）導入ガイド

## はじめに：本ガイドの使い方

このガイドは、組織における記録管理システム（MSR）を段階的に導入するためのアクションプランです。

### 導入の4つのステップ

**ステップ1** → 保存年限表のデジタル対応（基盤整備）  
**ステップ2** → 未管理デジタル記録の追加（範囲拡大）  
**ステップ3** → ハイブリッド管理プロセスの設計（運用確立）  
**ステップ4** → 自動化の実装（効率化）

各ステップは順番に実施することを推奨しますが、組織の状況に応じて並行実施も可能です。

## ステップ1：保存年限表をデジタル対応版に拡張

### このステップの目的

既存の紙ベースの文書保存年限表を、デジタル記録にも対応できる形式に拡張します。これにより、紙と電子を統一的に管理する基盤を整備します。

### 実施内容

**アクション1-1：現行の保存年限表を収集・整理する**

まず、各部門が独自に管理している保存年限表を集めます。

- 各部門の保存年限表を収集（担当：総務部）
- 重複・矛盾を整理して統合版を作成（担当：総務部）
- 法的根拠を再確認（担当：法務部・総務部）

成果物：部門別保存年限表一覧、統合保存年限表（暫定版）、法的根拠一覧表

**アクション1-2：デジタル記録対応の項目を追加する**

既存の保存年限表に、以下の5つの項目を追加します。

| 追加項目 | 内容 | 記入例 |
|---------|------|--------|
| 記録ID | 一意の識別子 | REC-L-001 |
| 媒体 | 紙/電子/両方 | 原本+PDF、電子のみ |
| 保管場所（電子） | システム名・サーバー名 | 契約管理システム、SharePoint |
| メタデータ要件 | 真正性確保のための要件 | 電子署名、タイムスタンプ、暗号化 |
| 廃棄方法（媒体別） | 紙と電子で異なる廃棄方法 | 紙：溶解、電子：DOD消去 |

**アクション1-3：拡張版テンプレートを作成する**

新しいテンプレートのフォーマット例：

```
記録ID | 記録種別 | 媒体 | 保存期間 | 根拠法令 | 保管場所（物理） | 保管場所（電子） | メタデータ要件 | 廃棄方法
```

**アクション1-4：パイロット部門で試行する**

1〜2部門を選定して、実際に拡張版フォーマットを使ってもらいます。

- パイロット部門を選定（担当：経営層）
- 選定部門の記録を拡張版フォーマットに記入（担当：パイロット部門）
- 運用上の課題を抽出（担当：総務部）
- フォーマットを修正して確定版を作成（担当：総務部）

成果物：拡張版保存年限表（パイロット版）、課題管理表、拡張版保存年限表（確定版）

## ステップ2：未管理デジタル記録を特定・追加

### このステップの目的

既存の文書管理ではカバーされていないデジタル記録（メール、チャット、会議録画など）を洗い出し、管理対象に追加します。

### 実施内容

**アクション2-1：デジタル記録の全社調査**

- 各部門が使用するデジタルツールの棚卸し（担当:IT部門）
- 業務で生成されるデジタル記録の種類を調査（担当:各部門）
- 現在の管理状況を確認（担当:IT部門・総務部）

成果物：デジタルツール一覧表、デジタル記録種別一覧、管理状況マップ

**アクション2-2：未管理デジタル記録を分類する**

発見された未管理デジタル記録を、以下のカテゴリーに分類します。

**コミュニケーション記録**
- 例：業務メール、チャット履歴、Web会議録画
- 現状の課題：個人管理、放置、削除基準なし

**業務システム記録**
- 例：CRM営業日報、勤怠記録、承認ワークフロー
- 現状の課題：保持期間不明、バックアップのみ

**制作物・成果物**
- 例：ソースコード、設計図、マーケティング資料
- 現状の課題：バージョン管理のみ、アーカイブなし

**外部発信記録**
- 例：公式SNS投稿、プレスリリース、ウェブサイト更新
- 現状の課題：未管理、削除されている可能性

**アクション2-3：優先順位付けと段階的追加**

すべてを一度に管理対象にするのは現実的ではないため、優先順位をつけて段階的に追加します。

**優先度：高**（法的義務あり、監査対象、契約関連）
- 例：契約関連メール、電子契約、電子帳簿

**優先度：中**（業務継続に必要、ナレッジ管理）
- 例：重要会議録画、プロジェクト文書、手順書

**優先度：低**（参考情報、一時的なコミュニケーション）
- 例：一般的なチャット、社内連絡メール

**アクション2-4：各記録種別の管理方針を決定する**

記録種別ごとに、保持期間、重要度判定基準、管理方法を決めます。

例：
- 業務メール → 3年（通常）/10年（契約関連）、メールアーカイブシステム
- チャット履歴 → 重要な意思決定のみ記録化、5年保存
- Web会議録画 → 経営会議・重要プロジェクト会議のみ、3年保存

**アクション2-5：統合記録保持スケジュールに統合する**

- 未管理デジタル記録を拡張版保存年限表に追加（担当：総務部・IT部門）
- 記録ID採番ルールを決定（担当：総務部）
- 管理方法・システムを明記（担当：IT部門）

成果物：統合記録保持スケジュール、記録ID採番ルール、システム対応表

## ステップ3：紙+電子のハイブリッド管理プロセスを設計

### このステップの目的

紙文書と電子記録を統合的に管理するプロセスを設計します。両方の媒体が混在する現実に対応した運用手順を確立します。

### 実施内容

**アクション3-1：ハイブリッド管理パターンを定義する**

組織内に存在する5つの管理パターンを明確にします。

1. **紙のみ**（例：押印が必要な契約書原本）
2. **電子のみ**（例：電子帳簿、メール）
3. **紙→電子移行**（例：スキャン保存した請求書）
4. **並行管理**（例：重要契約書の原本+PDF）
5. **電子→紙出力**（例：システム生成帳票）

それぞれのパターンについて、管理のポイントを整理します。

**アクション3-2：ライフサイクル各段階でのハイブリッド管理を設計する**

記録のライフサイクル（作成・捕捉、保存・利用、廃棄）の各段階で、紙と電子をどう扱うかを設計します。

**作成・捕捉段階**
- 紙文書：文書作成、押印 → 文書管理台帳に手入力 → 文書分類体系に従う
- 電子記録：システム入力、電子署名 → 自動メタデータ付与 → 同じ分類体系を適用
- ハイブリッド対応：作成媒体を記録、統合台帳への登録、共通の分類コード使用

**保存・利用段階**
- 紙文書：書庫・金庫 → 物理的な鍵管理 → 台帳から手動検索
- 電子記録：ファイルサーバー・クラウド → システム権限管理 → システム検索
- ハイブリッド対応：保管場所を両方記録、統一されたアクセス権限ポリシー、統合検索インターフェース

**廃棄段階**
- 紙文書：台帳から抽出 → 廃棄承認書（紙） → シュレッダー・溶解 → 廃棄記録簿（紙）
- 電子記録：システムアラート → ワークフロー承認 → 論理削除・物理破壊 → システムログ
- ハイブリッド対応：統合廃棄候補リスト、統一承認プロセス、媒体別廃棄記録、統合廃棄証跡台帳

**アクション3-3：ハイブリッド管理手順書を作成する**

手順書には以下の7つのセクションを含めます。

1. 目的と適用範囲
2. 用語の定義（紙文書、電子記録、原本、正本の定義）
3. 役割と責任
4. 作成・登録手順
5. 保存・利用手順
6. 廃棄手順
7. 監査と改善

**アクション3-4：移行プロセスを設計する**

既存の記録をハイブリッド管理に移行するプロセスを設計します。

**既存紙文書の電子化**
- 対象選定 → スキャン → メタデータ付与 → 原本処理判断
- 注意点：電帳法要件、法的証拠力の確認

**既存電子記録の体系化**
- 散在データの収集 → 分類 → 統合台帳登録
- 注意点：重複排除、バージョン管理

成果物：ハイブリッド管理手順書

## ステップ4：電子記録の自動捕捉とメタデータ付与

### このステップの目的

電子記録を業務システムから自動的に捕捉し、必要なメタデータを付与する仕組みを構築します。これにより、記録登録の現場負荷を大幅に軽減します。

### 実施内容

**アクション4-1：自動捕捉対象システムを選定する**

優先度をつけてシステムを選定します。

**優先度：高**
- 基幹業務システム（ERP、会計システム）
- 契約管理システム（電子契約、契約DB）
- 人事システム（人事DB、勤怠システム）

**優先度：中**
- 文書管理システム（SharePoint、Box）
- コミュニケーションツール（メール、Teams、Slack）

**アクション4-2：自動捕捉のアーキテクチャを設計する**

3つの連携パターンから、システムの特性に応じて選択します。

**パターンA：API連携方式**
```
業務システム → API → 記録管理システム → メタデータ付与 → アーカイブストレージ
```
- トリガー：レコード作成時、ステータス変更時、承認完了時
- データ取得：REST API、Webhook、バッチ処理
- 変換処理：JSON/XML → 標準フォーマット変換

**パターンB：ファイル連携方式**
```
業務システム → ファイル出力 → 監視フォルダ → 自動取込 → メタデータ付与
```
- 出力頻度：日次バッチ、週次バッチ
- ファイル形式：CSV、PDF、TIFF
- 監視・取込：スケジューラ、ファイル監視サービス

**パターンC：ユーザー操作方式**
```
ユーザー → 記録化ボタン → 確認画面 → メタデータ入力 → 記録登録
```
- UI配置：業務システム内にボタン埋込、ブラウザ拡張
- 入力項目：最小限（自動付与できない項目のみ）
- 承認フロー：必要に応じて承認ワークフロー追加

**アクション4-3：メタデータ自動付与を設計する**

メタデータを「自動付与できるもの」と「手動入力が必要なもの」に分類します。

**自動付与できるメタデータ**
- 作成日時（取得元：システムログ）
- 作成者（取得元:ユーザー情報）
- 記録種別（取得元：分類ルール）
- 保管場所（取得元：システム設定）
- ファイル形式（取得元：ファイル情報）

**手動入力が必要なメタデータ**
- 業務文脈（プルダウン選択）
- 重要度（レベル選択：高/中/低）
- 保持期間（記録種別から自動算出または手動入力）
- 関連記録（検索・選択）

**メタデータ検証ルール**
- 必須項目チェック：作成日時、作成者、記録種別は必須
- 形式チェック：日付形式、文字数制限
- 整合性チェック：保持期間と記録種別の整合性

**アクション4-4：段階的に実装する**

**Phase 1：パイロット実装（1システム）**
- 対象システム選定（担当：IT部門・記録管理責任者）
- API/連携方式の技術検証（担当：IT部門）
- メタデータ自動付与ロジックの開発（担当：IT部門）
- パイロット運用・検証（担当：IT部門・パイロット部門）

成果物：対象システム決定書、技術検証報告書、自動付与プログラム、運用検証報告書

**Phase 2：水平展開（複数システム）**
- 優先順位に基づく展開計画策定（担当：IT部門）
- システムごとの連携開発（担当：IT部門）
- メタデータ標準の全社統一（担当：記録管理責任者）
- 全システム稼働・監視（担当：IT部門）

成果物：システム展開計画、連携プログラム群、メタデータ標準仕様書、システム稼働報告書

**アクション4-5：運用・監視の仕組みを確立する**

以下の4つの項目を継続的に監視します。

- 自動捕捉成功率（目標：95%以上）
- メタデータ欠損率（目標：5%未満）
- システムエラー（即時対応）
- ストレージ使用量（80%到達で拡張検討）

## 付録：よくある質問

**Q1. 4つのステップを並行して進めてもいいですか？**

A. 推奨は順番通りですが、組織の状況によっては並行実施も可能です。ただし、ステップ1（保存年限表の拡張）は他のステップの基盤となるため、最優先で着手してください。

**Q2. パイロット部門はどのように選べばいいですか？**

A. 以下の基準で選定することを推奨します。
- 記録の種類が多様で代表的な部門
- 変化に柔軟で協力的な部門
- 他部門への影響力がある部門

**Q3. 自動化（ステップ4）にはどのくらいの期間がかかりますか？**

A. パイロット実装に3〜6ヶ月、全社展開に6〜12ヶ月が目安です。システムの複雑さと対象システム数によって変動します。

**Q4. 既存の紙文書はすべて電子化する必要がありますか？**

A. いいえ。電子化の必要性は、記録の重要度、利用頻度、保存期間などから判断します。すべてを電子化するのではなく、優先順位をつけて段階的に進めることを推奨します。
