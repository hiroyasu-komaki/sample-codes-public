# 記録管理（MSR）導入編

*MSR（Management System for Records）= ISO 30301

## 導入の2つのシナリオ

記録管理を導入する際、多くの組織では既に何らかの管理の仕組みが存在しています。既存の資産を最大限活用することで、効率的に記録管理を構築できます。

本編では、日本企業で最も多い2つの導入シナリオを紹介します：

| 項目 | シナリオ1: レガシー文書管理からの継承 | シナリオ2: ISMSとの統合 |
|------|-------------------------------------|----------------------------------|
| **対象組織** | • 紙ベースの文書管理規程が既に存在<br>• 文書保存年限表が整備されている<br>• 部門別の文書管理責任者が配置されている | • ISO 27001に基づくISMSを既に導入済み<br>• 情報資産台帳が整備されている<br>• セキュリティ管理体制が確立している |
| **活用できる既存資産** | ✅ 文書保存年限表 → 記録保持スケジュールのベース<br>✅ 文書分類体系・廃棄記録簿 → そのまま継承<br>✅ 部門別文書管理責任者 → 記録管理責任者へ | ✅ 組織体制<br>✅ 内部監査<br>✅ アクセス制御<br>✅ 暗号化<br>✅ バックアップ手順 |
| **必要な拡張・追加** | ⚠️ 紙+電子のハイブリッド管理への対応<br>⚠️ メタデータ要件の明記、電子データ削除の監査証跡強化<br>➕ デジタル記録（メール、チャット、Web会議録画等）の追加 | ⚠️ 情報資産台帳に「保持期間」「廃棄基準」「業務文脈」「法的根拠」を追加<br>⚠️ リスク評価に記録管理の観点を追加<br>➕ 記録保持スケジュール<br>➕ 廃棄プロセス・証跡管理<br>➕ 業務文脈分析書 |

---
<br>

## 各シナリオのメリット・デメリット

| 項目 | シナリオ1: レガシー文書管理からの継承 | シナリオ2: ISMSとの統合 |
|------|-------------------------------------|----------------------------------|
| **メリット（Pros）** | ✅ 現場に馴染みのある文書管理の延長で導入しやすい<br>✅ 保存年限表など既存ルールをベースにできる<br>✅ 部門責任者が既に配置されている<br>✅ 紙文書の管理ノウハウを活用できる | ✅ PDCAサイクルの基盤が既に確立されている<br>✅ 内部監査・マネジメントレビューの仕組みを流用できる<br>✅ セキュリティ管理との統合により効率的<br>✅ ISO認証の統合審査が可能<br>✅ IT部門の協力が得やすい |
| **デメリット（Cons）** | ⚠️ デジタル記録管理のノウハウが不足している可能性<br>⚠️ IT部門との連携体制を新規構築する必要がある<br>⚠️ システム化の投資が必要<br>⚠️ 紙中心の文化からの意識改革が必要 | ⚠️ セキュリティ観点のみで記録管理の本質（業務文脈、法的要件）が見落とされやすい<br>⚠️ 保持期間・廃棄基準の考え方が新規<br>⚠️ ISMS責任者の負担増加<br>⚠️ 記録管理特有の成果物を新規作成する必要がある |
| **このシナリオを選ぶべき組織** | • 製造業、建設業など紙文書中心の業界<br>• ITシステム化が進んでいない組織<br>• 総務・文書管理部門が強い組織<br>• 段階的にデジタル化を進めたい組織 | • IT・金融・通信など情報セキュリティが重要な業界<br>• 既にISO 27001認証を取得済み<br>• デジタル記録が業務の中心<br>• IT部門が強く、システム統合を推進できる組織 |

## 選択基準のチェックリスト

自組織に適したシナリオを判断するために、以下の質問に答えてください：

| チェック項目 | シナリオ1向き | シナリオ2向き |
|------------|------------|------------|
| ISO 27001認証を取得済みか？ | いいえ | **はい** |
| 情報資産台帳が整備されているか？ | いいえ | **はい** |
| 文書保存年限表が整備されているか？ | **はい** | いいえ |
| 部門別文書管理責任者が配置されているか？ | **はい** | いいえ |
| 業務の中心はデジタル記録か紙文書か？ | 紙文書中心 | デジタル記録中心 |
| IT部門の記録管理への関与度は？ | 低い | **高い** |
| 主導部門は？ | 総務・文書管理部門 | IT・セキュリティ部門 |

**判定:**
- シナリオ1向きが多い → レガシー文書管理からの継承アプローチ
- シナリオ2向きが多い → ISMSとの統合アプローチ
- 両方が混在 → ハイブリッドアプローチ（両シナリオの要素を組み合わせる）

#### シナリオ1
## レガシー文書管理からの継承

日本企業では**紙ベースの文書管理規程**が既に存在し、保存年限表（記録保持スケジュール）も整備されていることが多いため、この既存資産を最大限活用することが記録管理導入を効率的に進めるための重要なポイントです。

### 成果物の活用区分

| 成果物 | 区分 | 対応内容 |
|--------|------|----------|
| **文書保存年限表** | ⚠️ 拡張 | デジタル記録の追加、メタデータ要件の明記 |
| **文書分類体系** | ⚠️ 拡張 | 電子記録（メール、チャット、ログ等）の分類追加 |
| **文書管理規程** | ⚠️ 拡張 | デジタル記録への対応、真正性要件の追加 |
| **保管場所台帳** | ⚠️ 拡張 | 電子ファイルサーバー、クラウドストレージの追加 |
| **廃棄記録簿** | ⚠️ 拡張 | 電子データ削除の記録、監査証跡の強化 |
| **文書管理台帳** | ⚠️ 拡張 | ISO 15489メタデータ要件への拡張 |
| **部門別文書管理責任者** | ✅ 継承 | デジタル記録への責任範囲拡大 |
| **記録保持スケジュール（統合版）** | ➕ 新規作成 | 紙+電子のハイブリッド対応版 |
| **未管理デジタル記録一覧** | ➕ 新規作成 | メール、チャット、Web会議等 |
| **メタデータ仕様書** | ➕ 新規作成 | 真正性確保のための要件定義 |
| **ハイブリッド管理手順書** | ➕ 新規作成 | 紙+電子統合管理プロセス |
---
<br>

#### アプローチ

### 1. 記録保持スケジュールの統合拡張

**既存文書管理の保存年限表**を**MSR対応版**に拡張します。

**拡張のポイント：**
- 媒体欄に「紙+電子」を明記（ハイブリッド管理）
- 電子保管場所を追加（システム名、サーバー名）
- メタデータ要件を追加（真正性確保のため）
- 廃棄方法を媒体別に分割

<br>

### 2. 未管理デジタル記録の特定と追加

既存の文書管理では**カバーされていない**デジタル記録を洗い出し、管理対象に追加します。

**対象となる記録：**
- 業務メール（個人管理、放置されているもの）
- チャット履歴（未管理）
- ウェブ会議録画（個人PC保存）
- 営業日報（デジタル）（CRM内、保持期間不明）
- ソースコード（Git管理のみ）
- SNS投稿（公式）（未管理）

### 3. 段階的移行計画

**Phase 1：既存文書管理の棚卸し**

| タスク | 成果物 | 責任者 |
|--------|--------|--------|
| 既存保存年限表の収集 | 全部門の保存年限表一覧 | 総務部 |
| 文書分類体系の確認 | 文書分類マスタ | 総務部 |
| 保管場所の現地調査 | 保管場所台帳（更新） | 各部門 |
| 廃棄実績の確認 | 過去3年の廃棄記録 | 総務部 |

**目的:** 現状把握と課題抽出

**Phase 2：MSR対応への拡張**

| タスク | 成果物 | 責任者 |
|--------|--------|--------|
| デジタル記録の洗い出し | 未管理デジタル記録一覧 | IT部門 |
| 保存年限表の拡張 | 統合記録保持スケジュール | 総務部・IT部門 |
| メタデータ要件の定義 | メタデータ仕様書 | 記録管理責任者 |
| ハイブリッド管理手順の策定 | 紙+電子統合管理手順書 | 総務部 |

**目的:** MSR対応の設計

**Phase 3：パイロット運用**

| タスク | 成果物 | 責任者 |
|--------|--------|--------|
| 試行部門の選定 | パイロット部門決定書 | 経営層 |
| 記録管理ツールの試験導入 | ツール評価レポート | IT部門 |
| 部門担当者の教育 | 教育実施記録 | 記録管理責任者 |
| 運用課題の抽出 | 課題管理表 | パイロット部門 |

**目的:** 実運用での検証と改善

|--------|--------|--------|
| 全社方針の策定 | 記録管理ポリシー | 経営層 |

**目的:** 全社への定着と継続的改善

#### シナリオ2
## ISMSとの統合

すでにISO 27001に基づいてISMSを導入済みの企業において、情報資産台帳（データベース）が整備されている場合、後からISO 30301（MSR）に基づいて記録管理を導入しようとするのであれば、既存のISMSのプロセスや成果物が十分利用可能です。

### 成果物の活用区分

| 成果物 | 区分 | 対応内容 |
|--------|------|----------|
| **組織体制・責任者** | ✅ 継承 | ISMS責任者に記録管理責任を追加、または新規任命 |
| **経営層コミットメント文書** | ✅ 継承 | 情報セキュリティ方針に記録管理への言及を追加 |
| **内部監査プログラム** | ✅ 継承 | 記録管理の監査項目を追加して統合運用 |
| **アクセス制御・暗号化** | ✅ 継承 | 記録管理の観点でもそのまま利用可能 |
| **バックアップ手順** | ✅ 継承 | 記録保存の観点でもそのまま利用可能 |
| **情報資産台帳** | ⚠️ 拡張 | 「保持期間」「廃棄基準」「業務文脈」「法的根拠」を追加 |
| **リスク評価表** | ⚠️ 拡張 | 記録管理の観点（法的証拠能力喪失、コンプライアンス違反）を追加 |
| **業務プロセスマップ** | ⚠️ 拡張 | 記録発生ポイントと記録種別を追加マッピング |
| **教育プログラム** | ⚠️ 拡張 | 記録管理（保持期間遵守・適切な廃棄）のモジュールを追加 |
| **記録管理ポリシー** | ➕ 新規作成 | 記録管理固有の原則（真正性・信頼性・完全性・利用性）を定義 |
| **記録保持スケジュール** | ➕ 新規作成 | 各記録の法的根拠・保持期間・廃棄基準の一覧表 |
| **業務文脈分析書** | ➕ 新規作成 | どの業務プロセスでどんな記録が発生するか |
| **廃棄プロセス手順書** | ➕ 新規作成 | 保持期間満了判定・廃棄承認・廃棄証跡記録の手順 |
---

<br>

#### アプローチ

### 1. 情報資産台帳の拡張

ISO 27001の情報資産台帳は**セキュリティ観点**が中心なので、MSRの**記録管理観点**を追加します。

**拡張のポイント：**

| 項目 | ISO 27001で管理 | MSRで追加必要 |
|------|----------------|-------------------|
| 資産分類 | 機密性・完全性・可用性 | 記録種別・業務文脈 |
| リスク | 漏えい・改ざん・喪失 | 法的証拠能力の喪失 |
| 管理策 | アクセス制御・暗号化 | 保持期間・廃棄基準 |
| メタデータ | 作成者・分類ラベル | 業務プロセス・法的根拠 |
| ライフサイクル | セキュリティ要件の変化 | 作成→保存→評価選別→廃棄 |

<br>

### 2. 記録保持スケジュールの新規作成

ISO 27001にはない記録管理特化の文書を作成します。

**作成が必要な文書：**
- **記録保持スケジュール**: 各記録の保存期間と廃棄基準の一覧表
- **業務文脈分析書**: どの業務プロセスでどんな記録が発生するか
- **評価選別基準**: 記録の保存価値をどう判断するか

<br>

### 3. 段階的移行計画

**Phase 1：既存ISMS基盤の確認**

| タスク | 成果物 | 責任者 |
|--------|--------|--------|
| 情報資産台帳の棚卸し | 全情報資産の一覧 | ISMS責任者 |
| セキュリティ管理プロセスの確認 | プロセス文書一覧 | ISMS事務局 |
| 内部監査・教育プログラムの確認 | 既存プログラム一覧 | 内部監査部門 |

**目的:** 現状把握と記録管理要件の特定

**Phase 2：MSR対応への拡張**

| タスク | 成果物 | 責任者 |

| 情報資産台帳への項目追加 | 拡張版情報資産台帳 | ISMS責任者・記録管理責任者 |

**目的:** MSR対応の設計

**Phase 3：パイロット運用**

| タスク | 成果物 | 責任者 |
|--------|--------|--------|
| 試行部門の選定 | パイロット部門決定書 | 経営層 |

**Phase 4：全社展開**

|--------|--------|--------|
| 記録管理ポリシーの策定 | 記録管理ポリシー | 経営層 |
| ISMS文書への統合 | 統合マネジメントシステム文書 | ISMS責任者 |

### 具体例

#### 例1：情報資産台帳の拡張

**既存（ISO 27001）：**
```
資産名：顧客契約データベース
分類：機密
対策：暗号化、アクセス制限
```

**追加（MSR）：**
```
記録種別：契約記録
法的根拠：商法522条
保持期間：10年（契約終了日起算）
廃棄方法：物理的破壊 + 記録簿への記載
業務文脈：営業部門の契約締結プロセス
```

**統合後：**
```
資産ID: CRM-001
資産名: 顧客契約データベース
分類: 機密（ISO 27001）
記録種別: 契約記録（MSR）
法的根拠: 商法522条
保持期間: 10年（契約終了日起算）
セキュリティ対策: 暗号化、アクセス制限
廃棄基準: 10年経過後、法務部長承認
業務文脈: 営業部門の契約締結プロセス
```

#### 例2：人事記録の統合管理

**ISO 27001の視点（既存）：**
- 個人情報として「機密」に分類
- アクセスは人事部門のみ
- 暗号化して保管

**MSRの視点（追加）：**
- 労働基準法により退職後3年保存
- 3年経過後は人事部長承認で廃棄
- 廃棄時は記録管理台帳に記載
- マイクロフィルム化の要否を評価

**統合台帳：**
```
資産ID: HR-001
資産名: 従業員人事ファイル
分類: 機密（ISO 27001）
記録種別: 人事記録（MSR）
保持期間: 退職後3年（労基法）
セキュリティ対策: 暗号化、アクセス制限
廃棄基準: 3年経過後、人事部長承認
```
